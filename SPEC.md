# Spec: פלטפורמת בקשות קהילתית — bakbookim

## 1. רקע ומטרה

מנהל האתר [bakbookim.com](https://bakbookim.com) (חנות אלכוהול קהילתית — רכישה ישירה מיבואנים/יקבים במחירים מוזלים, מבוססת קבוצות וואטסאפ ונקודות איסוף מתנדבים) מקבל כיום מאות עד אלפי בקשות ביום דרך וואטסאפ — הן בקשות להחזרת מוצרים שהיו בעבר בפרויקט, והן בקשות למוצרים חדשים שטרם הוצעו.

**המטרה:** לבנות פלטפורמת web נפרדת שמרכזת את כל הבקשות, מקבצת אוטומטית בקשות דומות (גם כשהן מנוסחות אחרת, גם כשהן בעברית/אנגלית מעורבבות), ומציגה רשימות סיכום ציבוריות ומעודכנות — כדי להחליף את הכאוס של קליטת בקשות ידנית בוואטסאפ.

**עיצוב:** צריך להיות בעל זיקה ויזואלית לאתר bakbookim.com (טונים, טיפוגרפיה, סגנון כללי) — יש לבדוק את העיצוב החי של האתר (devtools/computed styles) ולהתאים בהתאם, לא רק להעריך מהזיכרון.

**שפה:** ממשק כולו בעברית, RTL. תוכן משתמשים (בקשות) יכול לערבב עברית ואנגלית — יש לתת לזה מענה מפורש (ראה סעיף 5).

---

## 2. שני מסלולים נפרדים (חשוב מאוד)

הפלטפורמה מנהלת **שתי רשימות ומסלולים נפרדים לחלוטין** — לא תת-קטגוריה אחת:

| | בקשה לחזרת מוצר | בקשה למוצר חדש |
|---|---|---|
| הגדרה | מוצר שכבר היה בפרויקט בעבר, אנשים רוצים שיחזור | מוצר שמעולם לא הוצע בפרויקט |
| טופס | Dropdown/autocomplete מבוסס clusters/היסטוריה קיימת | טקסט חופשי + dropdown קטגוריה בלבד |
| שדה "למה" | לא רלוונטי | כן, אופציונלי (ראה סעיף 6) |
| קיבוץ (clustering) | בתוך cluster-pool נפרד | בתוך cluster-pool נפרד |
| רשימה ציבורית | רשימה עצמאית | רשימה עצמאית |

**חשוב:** `request_type` (`return` / `new`) הוא לא רק תווית — הוא partition בשלב הקיבוץ. בקשות משני הסוגים לעולם לא יתקבצו יחד גם אם הטקסט זהה במקרה.

## 3. זרימת נתונים — סקירה כללית

```
[טופס ציבורי] 
     ↓ POST /requests
[שמירת בקשה גולמית] 
     ↓ (מיידי, סינכרוני או queue קליל)
[שלב נרמול AI] → מחלץ category / brand / variant / vintage + מנרמל שם לצורה קנונית אחידה
     ↓
[חישוב embedding] (מודל רב-לשוני)
     ↓
[השוואה ל-clusters קיימים מאותו request_type] (pgvector cosine similarity)
     ↓
[הצטרפות ל-cluster קיים / יצירת cluster חדש] → מעדכן cluster_daily_counts

--- כל 24 שעות (scheduled job) ---
[job יומי] → מרענן snapshot מסוכם + מריץ AI summary לקלאסטרים עם reason-ים
     ↓
[רשימות ציבוריות קוראות מה-snapshot, לא מהטבלאות הגולמיות]

--- פאנל ניהול ---
[קורא נתונים חיים מהטבלאות הגולמיות, לא מה-snapshot]
```

## 4. סכימת בסיס הנתונים (PostgreSQL + pgvector, Neon)

### `requests`
בקשה גולמית בודדת, כפי שהוגשה.

| שדה | טיפוס | הערות |
|---|---|---|
| id | UUID / serial | PK |
| request_type | enum(`return`, `new`) | חובה |
| category | text | וויסקי / יין / ג'ין וכו', dropdown סגור |
| original_text | text | הטקסט הגולמי כפי שהמשתמש הקליד |
| canonical_brand | text, nullable | מולא ע"י שלב הנרמול |
| canonical_variant | text, nullable | גרסה/טעם/שנה אם צוין |
| reason | text, nullable | רק ל-`request_type = new`, אופציונלי |
| submitter_name | text, nullable | אופציונלי, **לא ציבורי** |
| submitter_phone | text, nullable | אופציונלי, **לא ציבורי** |
| is_joined_existing | boolean | true אם הוגש דרך "גם אני רוצה" על cluster קיים |
| cluster_id | FK → clusters.id | |
| embedding | vector | pgvector |
| ip_hash | text | hash בלבד, לצורך rate limiting — לא IP גולמי |
| created_at | timestamp | |

### `clusters`
קבוצת בקשות שזוהו כאותו מוצר, בתוך אותו `request_type`.

| שדה | טיפוס | הערות |
|---|---|---|
| id | UUID / serial | PK |
| request_type | enum(`return`, `new`) | חובה — חלק מהמפתח הלוגי |
| category | text | |
| canonical_name | text | השם המוצג ברשימות הציבוריות — אחיד, קריא, תואם ל-bakbookim |
| status | enum(`active`, `fulfilled`) | מנוהל ע"י המנהל בפאנל |
| ai_summary_note | text, nullable | מחושב ב-job היומי, תמצות של ה-reason-ים (רק ל-`new`) |
| total_requests | int | מונה מצטבר, מתעדכן בזמן אמת |
| unique_submitters | int | ספירת מספרי טלפון/שמות ייחודיים (best-effort, לא כל בקשה כוללת פרטים) |
| first_seen_at | timestamp | |
| last_seen_at | timestamp | |

### `cluster_daily_counts`
טבלת ספירה יומית — מאפשרת סיכום מהיר לכל טווח זמן בלי לסרוק את כל הבקשות הגולמיות.

| שדה | טיפוס | הערות |
|---|---|---|
| cluster_id | FK → clusters.id | |
| date | date | |
| count_that_day | int | מספר הבקשות שנוספו לאותו cluster באותו יום |

PK מורכב: (cluster_id, date)

### `daily_snapshots`
תוצר ה-job היומי — מה שהציבור בפועל צורך.

| שדה | טיפוס | הערות |
|---|---|---|
| id | serial | PK |
| snapshot_date | date | |
| request_type | enum(`return`, `new`) | |
| range_type | enum(`week`, `month`, `3months`, `6months`, `year`) | חושב מראש לכל טווח כדי שתצוגת האתר תהיה מהירה |
| ranked_list | JSON | מערך של עד 30 clusters, ממוינים, עם total_requests + unique_submitters |
| created_at | timestamp | |

### `admin_users`
| שדה | טיפוס |
|---|---|
| id | serial |
| username | text |
| password_hash | text |

## 5. טיפול בעברית/אנגלית מעורבבת

**הבעיה:** משתמשים כותבים "וויסקי מקאלן 18", "Macallan 18 whiskey", "מקאלן 18 שנה" — כל אלה חייבים להתקבץ יחד, למרות הבדלי שפה ותעתיק.

**הפתרון (שני שלבים, לא embedding בלבד):**

1. **שלב נרמול AI (חובה, לפני embedding):** קריאה ל-Claude API שמחלצת מהטקסט הגולמי:
   ```json
   {
     "category": "וויסקי",
     "canonical_brand": "Macallan",
     "canonical_variant": "18",
     "specificity": "specific_bottle"
   }
   ```
   הפרומפט צריך להנחות את המודל לנרמל שמות מותגים לצורה קנונית אחידה (בדרך כלל תעתיק/שם כפי שמופיע בפועל על הבקבוק או באתר bakbookim), **בלי קשר לשפה שבה המשתמש כתב**.

2. **קיבוץ ראשוני לפי שדות מנורמלים** (התאמה מדויקת/כמעט-מדויקת של category+brand+variant בתוך אותו request_type), ורק בתוך אותה קבוצה משתמשים ב-embedding כדי לתפוס ניסוחים/תעתיקים נוספים.

3. **מודל embedding:** יש להשתמש במודל התומך רב-לשוניות אמיתית (Claude / OpenAI embeddings) — לא מודל חד-לשוני.

**שני שדות נשמרים תמיד:** `original_text` (לתיעוד ולפאנל, לא נערך) + `canonical_name` ברמת ה-cluster (מוצג לציבור, שם אחד עקבי לכל מוצר).

## 6. שדה "הסבר" בבקשות מוצר חדש

רלוונטי רק ל-`request_type = new`.

**בטופס:**
```
שדה: reason (טקסט חופשי, אופציונלי — לא חובה)
Placeholder: "נשמח שתכתבו למה אתם רוצים את המוצר הזה — 
זה עוזר לשאר חברי הקהילה להכיר מוצרים חדשים 😊"
```

**ברשימה הציבורית הראשית:** לא מוצג טקסט חופשי — רק שם מוצר, קטגוריה, מונה בקשות. נשארת נקייה.

**תצוגה מורחבת (על-פי דרישה):** ליד כל פריט — כפתור/אייקון "למה מבקשים את זה? 💬". בלחיצה נפתחת הרחבה שמציגה:
1. `ai_summary_note` — תמצות AI קצר של הסיבות הנפוצות (מחושב ב-job היומי אם יש reason-ים ל-cluster)
2. מתחתיו — 3-5 מהתגובות הגולמיות האחרונות/המפורטות, עם אופציית "טען עוד"

---

## 7. פיצ'ר "גם אני רוצה" (Join Existing)

ברשימה הציבורית (חזרה או חדש), ליד כל cluster — כפתור **"גם אני רוצה את זה 🙋"**.

- לוחצים → פותח מיני-פורם (רק שם/טלפון אופציונליים, בלי לחזור על כל הטופס)
- שולח `POST /requests/join/{cluster_id}`
- יוצר `requests` record עם `is_joined_existing = true`, מקושר ישירות ל-cluster הקיים (בלי מעבר בשלב הנרמול/embedding — כבר ידוע לאיזה cluster זה שייך)
- מעלה את `total_requests` ואת `cluster_daily_counts` של היום

**חשוב:** זה גם UX נוח יותר, וגם משפר את איכות הדאטה — כי חלק מהבקשות לא צריכות בכלל לעבור את "צינור" הקיבוץ האוטומטי.

---

## 8. תיוג "סופק" (Fulfilled)

בפאנל הניהול, לכל cluster יש אפשרות לשנות `status` ל-`fulfilled`.

ברשימה הציבורית, cluster שסומן `fulfilled` מקבל תג ויזואלי בולט (למשל "✅ הגיע לפרויקט!"), ונשאר ברשימה (לא נמחק) — כדי לתת לקהילה משוב שהבקשות שלהם באמת משפיעות.

---

## 9. רשימות ציבוריות — פילטרים ותצוגה

**כללי:**
- שתי רשימות עצמאיות: "בקשות לחזרה" ו"בקשות חדשות" (טאבים/דפים נפרדים)
- עד **30** פריטים ברשימה (config, לא hardcoded — קל לשינוי בעתיד)
- מתעדכן פעם ב-24 שעות (מבוסס `daily_snapshots`)

**אפשרויות מיון/פילטר:**
1. **הכי מבוקש** (ברירת מחדל) — לפי `total_requests`, עם טוגל למיון לפי `unique_submitters` (כדי לא לתת משקל יתר למי ששלח בקשה כפולה כמה פעמים)
2. **טרנדינג** — קצב עלייה (delta בין תקופות, מבוסס `cluster_daily_counts`)
3. **חדש לגמרי** — clusters שנפתחו לראשונה בטווח הזמן הנבחר (רלוונטי בעיקר למסלול "מוצר חדש")
4. **קטגוריה** — סינון לפי וויסקי/יין/ג'ין וכו'
5. **טווח זמן** — שבוע / חודש / 3 חודשים / חצי שנה / שנה (חוצה את כל הפילטרים האחרים; נשלף מ-`daily_snapshots` לפי `range_type` המתאים)

## 10. פאנל ניהול

**גישה:** לא מקושר בניווט הציבורי, נתיב לא-מפורסם (למשל `/admin`), חי על אותו דומיין, מוגן ב-username+password (auth פשוט, ללא מערכת הרשאות מדורגת בשלב זה).

**מסכים:**
1. **טבלת בקשות/קלאסטרים** — עיצוב "ידידותי ל-Excel", לא ממשק SQL:
   - תיבת חיפוש חופשית אחת (מחפשת גם טלפון, גם שם, גם טקסט מקורי, גם canonical_name)
   - עמודות: תאריך, שם מוצר (canonical), טקסט מקורי, שם, טלפון, request_type, status
   - פעולות לכל שורה: עריכה, מחיקה
2. **ניהול clusters** — סימון `fulfilled`, אפשרות מיזוג/פיצול ידני של קלאסטרים (למקרה שה-AI טעה — למשל קיבץ "וויסקי" עם "בורבון" בטעות)
3. **דשבורד קצר** (אופציונלי, לא MVP קריטי) — מספרי סיכום: כמה בקשות היום/השבוע, כמה משתמשים ייחודיים

## 11. הגנת ספאם / קצב

**MVP:** Rate limiting פשוט לפי IP (למשל: מקסימום X בקשות בשעה מאותו IP). ללא CAPTCHA בשלב זה — ניתן להוסיף בהמשך אם יתגלה צורך.

`ip_hash` נשמר (לא IP גולמי) לצורך אכיפת ההגבלה מבלי לשמור מידע מזהה מיותר.

## 12. API Endpoints (טיוטה)

```
POST   /requests                     — הגשת בקשה חדשה (תומך בכמה מוצרים בבקשה אחת, ראה סעיף 13)
POST   /requests/join/{cluster_id}   — הצטרפות מהירה ("גם אני רוצה")

GET    /lists/{request_type}         — רשימה ציבורית (return / new)
                                        query params: sort, category, range

GET    /admin/requests               — טבלת בקשות (עם חיפוש/פילטר)
PATCH  /admin/requests/{id}
DELETE /admin/requests/{id}

GET    /admin/clusters
PATCH  /admin/clusters/{id}          — כולל שינוי status, מיזוג/פיצול
POST   /admin/login
```

## 13. טופס — בקשה עם כמה מוצרים

טופס תומך בכפתור "הוסף מוצר נוסף" — מאפשר למשתמש להוסיף כמה שדות מוצר באותה הגשה. בשרת, כל מוצר מתפרק לרשומת `requests` נפרדת (embedding/cluster נפרדים לכל אחד) — כדי לשמור על דיוק הקיבוץ. חוויית משתמש אחידה, נתונים נקיים מאחורי הקלעים.

## 14. תשתית ופריסה (Deployment)

| רכיב | פלטפורמה | סיבה |
|---|---|---|
| Backend (FastAPI) | **Railway** | דיפלוי אוטומטי מ-GitHub, תומך ב-scheduled jobs (ה-job היומי), עומד בעומס של אלפי בקשות ביום |
| DB | **Neon** (Postgres + **pgvector**) | מנוהל, free tier נדיב, כבר בשימוש בפרויקטים קודמים |
| Frontend (React) | **Vercel** | דיפלוי אוטומטי, Preview Deployments לכל PR — מאפשר לשתף לינק בדיקה עם מנהל הפרויקט לפני מיזוג ל-production |
| מקור | **GitHub** | Repo מרכזי, כל push/PR מפעיל דיפלוי בהתאם |

**זרימת עבודה מוצעת:** עבודה לוקאלית עם Claude Code → branch → PR ב-GitHub → Vercel/Railway preview אוטומטי → שיתוף לינק עם מנהל הפרויקט לבדיקה → מיזוג ל-main → דיפלוי production אוטומטי.

## 17. סיכום החלטות מרכזיות

- שני מסלולים נפרדים לגמרי: חזרה / חדש — לא תת-קטגוריה
- קיבוץ דו-שכבתי: נרמול AI → embedding, לא embedding בלבד
- רב-לשוניות מטופלת ע"י נרמול לשם קנוני + מודל embedding רב-לשוני
- Snapshot יומי מבוסס ספירה יומית לכל cluster — תומך בכל טווחי הזמן בזול חישובית
- Top 30 (config)
- "גם אני רוצה" + תיוג "סופק" — שתי תוספות קהילתיות ל-MVP
- פאנל ניהול נגיש מהאתר (לא ציבורי) עם ממשק טבלה/חיפוש ידידותי, לא SQL
- Rate limiting בלבד ל-MVP, בלי CAPTCHA
- Railway + Neon (pgvector) + Vercel + GitHub
